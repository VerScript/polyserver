Jules Daily Run Findings

1. Potential NULL pointer dereference in `verscript_src/src/main.c`. The `set_var()` function can return `NULL` when `MAX_VARS` is exceeded, but the code was not checking for this before dereferencing the variable pointer in `prompt` and identifier assignment sections. I added `if (v)` null checks to fix this.
2. Memory leaks in `verscript_src/src/main.c`. Tokens peeked using `peekToken()` were dynamically allocating memory for their `value` strings, but these strings were not being freed in `evaluate_expression` and the identifier assignment logic in the main parsing loop. I added `free()` calls to clean up the token values.
- Fixed compiler warning (unused 'is_string' variable) in 'verscript_src/src/main.c'
- Added division by zero error handling to 'evaluate_expression' in 'verscript_src/src/main.c'
- Prevented command injection risk by switching from 'exec' to 'execFile' in 'services/VS-Sharp/server.js'
- Fixed unhandled ftell return value error in `verscript_src/src/main.c` that caused crashes when executing directories.
- Fixed memory leaks during variable reassignment in `verscript_src/src/main.c` where string variables dynamically allocated in the symbol table were not freed when reassigned to an int, and memory was not cleaned up on program exit.
- Fixed `fetchGitHubBinary` in `polyserver.js` where the `headers` argument was improperly passed, causing binary files to download as JSON metadata.
- Fixed uninitialized memory bug in file reading in `verscript_src/src/main.c` to use `fread` return value for the null terminator.
- Fixed symbol table exhaustion vulnerability in `verscript_src/src/main.c` due to `get_var` using `strcmp` against a truncated string. Switched to `strncmp(..., 63)`.
